# `/jr-skill-audit` — Edge cases

**Reference material** for `/jr-skill-audit`, loaded on demand — NOT read at Phase 1 Track A (these are case→behavior mappings, not load-bearing procedure, so no hard-fail guard applies). `jr-skill-audit/SKILL.md`'s `## Edge cases` section points here. Update here to update the edge-case reference.

| Case | Behavior |
|------|----------|
| Cache > 7 days old, network unavailable | Use stale cache + `[STALE: cached YYYY-MM-DD]` warning. `feature-adoption-reviewer` runs but tags every finding `[Source: cached YYYY-MM-DD]`. |
| Cache missing, network unavailable | Skip `feature-adoption-reviewer` with a Phase 7 warning. Other 5 dimensions run normally. |
| Bare `/jr-skill-audit` with zero skills installed | `[ABORT — UNMATCHED SCOPE]` with the fully-unfiltered wording ("nothing is auditable in either scope"): no flag is named (none was passed) and no "Available skills" hint fires, since that hint is bare-positional-only. Should never trigger in practice — `/jr-doctor` would already flag a broken skills install. |
| Skill with `model:` field referencing a model alias added after this skill was last updated | `frontmatter-reviewer` cites the live skills-doc model section: if the alias is in the doc, finding is dropped; if not, flagged as `WARN_MODEL`. |
| Skill referencing a `shared/<name>.md` that exists but was renamed | `shared-drift-reviewer` flags as broken-ref (mirrors `/jr-doctor` Group I). Cross-references the canonical name if the renamed file is found via `git log --diff-filter=R`. |
| Skill with `disable-model-invocation: true` AND a non-empty `description` exceeding 1,024 chars | `frontmatter-reviewer` flags as `medium`: when DMI is true, the description isn't loaded into context, so verbose descriptions are dead weight in `/`-menu listings. |
| `--plugin=<name>` (plugin audit) | Resolves to the plugin's git-backed marketplace clone (`~/.claude/plugins/marketplaces/<mp>/<source>/skills/`); audits git-tracked skills only, all 7 dimensions, every finding tagged `[third-party — verify against plugin docs]`. Read-only/advisory — owned by the plugin author. |
| `--plugin=<name>` not declared by any marketplace | `[ABORT — UNMATCHED SCOPE]`: `Plugin <name> not found. Available plugins: <union of plugins[].name>.` |
| `--plugin=<name>` whose marketplace is not a git repo (e.g. `claude-plugins-official`) | Warn and skip (`… non-git marketplace …; skill-audit only audits git-versioned skills.`); abort if it was the sole resolution. |
| `--plugin=<name>` with an external object-form `source` (`git-subdir`/`url`, e.g. `pensyve`) | Warn and skip at Track B step 1 (`… uses an external (git-subdir/url) source, not an in-marketplace path …`); abort if it was the sole resolution. Vendored-elsewhere plugins aren't reachable via the marketplace clone. |
| Same plugin `<name>` declared by two marketplaces | Audit both; the `marketplace` tag disambiguates findings. |
| Bare `/jr-skill-audit` from a foreign repo that has its own skills (e.g. `~/myproject` with `.claude/skills/`) | Auto-scope: audits that repo's skills ALONE. Personal is still enumerated (so shadow detection fires) but not audited. The Phase 1 `Scope:` line and the Phase 7 `Roots:` line both name the chosen root. `--scope-only=both` restores the pre-auto behavior. |
| Bare `/jr-skill-audit` from a foreign repo with NO `.claude/skills/` in the walk (e.g. `~/Documents`, or any non-repo dir) | Auto-scope evaluates to `both` — the `projectRoots`-non-empty conjunct fails, so it does NOT narrow. Result: personal-only audit, no abort. |
| Bare `/jr-skill-audit` from a foreign repo whose `.claude/skills/` **exists but yields no auditable skills** (empty dir, only dirs without a `SKILL.md`, or every skill gitignored) | The `projectRoots` conjunct tests directory *existence*, so auto-scope proposes `project` and personal is dropped — then enumeration/gitignore empties the set. The **auto-narrow fallback** restores the personal candidates (running gitignore exclusion on them, which the scope filter skipped), sets `effectiveScope=personal` + `autoNarrowed=false` + `autoNarrowFallbackFired=true`, and the run proceeds as a personal-only audit. No abort, and no `[auto-scoped to project]` qualifier — the run was not narrowed after all. It does print the fallback `Scope:` line (gated on `autoNarrowFallbackFired`), which is the only record that the project root was tried and yielded nothing; without it this run would be byte-identical to one where auto-scope never applied. Auto-narrowing is a cost optimization, never a failure mode. |
| Auto-narrow fallback fires but BOTH scopes are empty (nothing auditable anywhere) | Falls through to `[ABORT — UNMATCHED SCOPE]` normally. The fallback un-narrows; it does not swallow a genuine "nothing to audit" case. Its step 3 sets `effectiveScope=both` on this arm (the one place the field means "what was tried", not "what was audited" — nothing was audited), so the `Roots:` line names both roots and every name in the message can be attributed to one. The abort message names no flag (none was passed) and branches on `excludedCandidates`: with exclusions it names the gitignored skills (the common case, since "empty" here usually means *emptied by* gitignore, including the personal candidates dropped in the fallback's own step 2); with none it says nothing is auditable in either scope. |
| `autoScope=project` computed, but a bare positional or `--scope=<glob>` is present | `effectiveScope=both`, `autoNarrowed=false` — the argument-set filter overrides the proposal. Display sites read `effectiveScope`/`autoNarrowed`, so neither the `Scope:` line nor the `Roots:` qualifier claims a narrowing that did not happen. This is why no site may read `autoScope` directly. |
| `/jr-skill-audit <name>` or `--scope=<glob>` from a foreign repo | Auto-scope does NOT apply — an argument-set filter is the user's own narrowing, and `<name>` stays resolvable in either scope. Without this carve-out, `/jr-skill-audit jr-audit` from a foreign repo would report "not found" for a personal skill that plainly exists. |
| `--scope-only=both` | Names the pre-auto-scope default explicitly: audit personal + project together. A no-op wherever `autoScope` is already `both` (the skills repo, a non-repo dir, a repo with no skills). |
| Bare `/jr-skill-audit` invoked from a tackle worktree | The worktree path counts as project scope. A worktree of a foreign repo auto-scopes to that worktree's `.claude/skills/` (and parents up to the worktree's repo root). A worktree of the skills repo itself has no `.claude/skills/`, so the non-empty conjunct fails → `both` → personal-only, as before. |
| CWD is `~` or under `~/.claude/skills/` | Project walk skips any dir whose realpath resolves under personal-root, preventing duplicate iteration. `projectRoots` is therefore empty → auto-scope is `both` → personal-only audit, no synthetic shadow findings. |
| `--scope-only=project` from a dir with no `.claude/skills/` anywhere in the walk | Empty target set → `[ABORT — UNMATCHED SCOPE]` per the canonical mapping. The abort message includes `--scope-only=project` so the user can drop the flag and retry. The explicit flag aborts where the auto-scope default would have fallen back to personal — pinning a scope means the user asserted it exists. |
| Same skill name in personal + project (deliberate per-repo override) | Lead emits `scope-resolution` finding with `clarify: true`. User picks "Drop this finding" in the Phase 4 [Clarify] flow. The finding is the ONLY collision-driven artifact — it never changes which files are audited. |
| Shadow exists but the shadowed side is not in the audited set (auto-scope, or an explicit `--scope-only`) | The finding still fires — shadow detection groups the *enumerated* set (pre-argument-filter, pre-scope-filter, pre-gitignore), not the surviving one. It anchors on the personal SKILL.md, which is readable regardless, so Phase 3's sanity-check passes. Consequence on an otherwise project-only run: the finding carries `scope: personal`, so `count(distinct scopes) > 1` and the Phase 7 `By scope:` line renders. That is honest — the run really did produce a finding about a personal file. |
| `--add-dir` directories at session start | Out of scope (v2 candidate). `/jr-skill-audit` has no access to the running session's `--add-dir` set. If a user's monorepo workflow depends on `--add-dir`-mounted skills, they should run `/jr-skill-audit` from each mount root separately. |
| Bare positional `<name>` with `--scope-only=personal` but `<name>` only in project | Abort per flag-conflict block: `Skill <name> not found in personal scope (exists in project: drop --scope-only or use --scope-only=project).` |
| `gh` CLI not installed or unauthenticated | Track C `claude-code-changelog` fetch fails. `feature-adoption-reviewer` falls back to skills-doc only (changelog evidence missing). Phase 7 warns. |
| Skill legitimately needs `opus` despite mechanical-looking phases | `model-routing-reviewer` sets `clarify: true` with a `clarificationQuestion` rather than asserting a finding — premium tier is often a deliberate headroom choice, and a hard finding would re-fire on every re-run. |
| A reviewer finding's `file` resolves (via `realpath`) outside the dispatched target set | Not Read — routed to `ACTION REQUIRED` (renders under `Audit integrity`) as a reviewer-quality issue, so a prompt-injected reviewer cannot steer the Phase 3 step-1 codeExcerpt Read to an arbitrary path. The lead-synthesised `scope-resolution` finding is exempt (gated on provenance, not the `dimension` field): its `file` legitimately anchors on a personal SKILL.md that may be enumerated-but-not-dispatched on an auto-narrowed run. |
| A reviewer finding's shared-form `source` (`~/.claude/skills/shared/<file>:<line>`) escapes the shared dir via `../` | Not opened — routed to `ACTION REQUIRED`. Phase 3 step 2 confirms the cited line only for a `<file>` that `realpath`-resolves inside `~/.claude/skills/shared/`, mirroring the step-1 `file` containment and the #68 step-2 `<skill>/SKILL.md` guard. |
| `--report`/`--report-path` write fails (permission denied, disk full, unwritable `--report-path`) | Non-fatal. Emit `Report file not written: <reason>. Console output above is the record.` and continue; the console report is the primary deliverable. A report-write failure alone never forces a non-zero exit (`protocols/report-write.md` "Failure is non-fatal"). |
| `--report` on a **project**-scope run, invoked from a subdirectory of the repo | The report path is `<repo-root>/.claude/skill-audit-<date>.md`, above CWD. `Write(.claude/skill-audit-*)` is CWD-anchored, so the write is not pre-authorised → Claude Code prompts once. Prompt-free only when invoked from the repo root, or when the user has `Write(/.claude/**)` in their project/user settings. |
| `--report` on a **project**-scope run under `--auto-approve`/headless | A not-pre-authorised in-repo write cannot prompt, so it is skipped (non-fatal, per above) unless the user has added `Write(/.claude/**)` to `.claude/settings.json`. Personal/both/plugin reports write prompt-free (absolute `~`-anchored grant), so CI archival works for those scopes without extra settings. |
| `--report-path=<path>` points inside a **different** git repo than the audited one | `report-write.md`'s gitignore probe keys on the *resolved path's* container repo, so enforcement fires there: warn-if-tracked, inform-with-glob-if-not-ignored (advisory; no `.gitignore` mutation). The write itself is not pre-authorised (relies on the user's settings or a prompt). |
| Second `--report` run, same scope, same UTC day | Overwrites the previous `skill-audit-<date>.md` (filename is date-only; the atomic `.tmp`→`mv` replaces in place), matching `/jr-audit`. The `Generated:` line inside carries the full ISO-8601 timestamp, so the surviving file is identifiable as the latest run. |
| `--plugin=<name> --report` — report on an untrusted third-party spec | The audited SKILL.md files are owned by the plugin author (untrusted) and the report writes out-of-repo (`~/.claude/skill-audit-reports/`), so gitignore-enforcement does not apply there and jr-skill-audit runs no secret-value redaction (loads no pattern catalog). A secret-shaped string in a third-party spec, if excerpted into a finding, would persist un-redacted in the report. Documented known limitation (scoped carve-out from `shared/display-protocol.md`); mitigate by reviewing the report before sharing it, or deleting it after. |
