# Personal/Project Scope Discovery (`/jr-skill-audit`, non-`--plugin` path)

**Canonical procedure** for the personal + project scope-root discovery, enumeration, filtering, and gitignore exclusion. Read into lead context at Phase 1 Track A **only when `--plugin` is NOT set** (the complementary conditional to `plugin-scope.md`, which is read only under `--plugin`; the two paths are mutually exclusive). Returns the surviving target set tagged `scope = personal | project`, consumed by the lead-side shadow detection and Phase 2 dispatch. Smoke-parse anchors enforced at the read site: `Scope roots` AND `Gitignore exclusion`.

**Scope roots** (computed before any enumeration) — personal/project path (when `--plugin` is NOT set):

- **`realpath` availability probe**: run `command -v realpath >/dev/null 2>&1` once and store `REALPATH_AVAILABLE=true|false`. When `false`, the skip conditions and dedupe step below fall back to literal trailing-slash path comparison (`case "$dir/" in "$personalRoot"/*) ;; esac`) instead of `realpath`-normalized paths — slightly less symlink-robust but functionally equivalent for the common case (direct duplicates still caught; symlink aliases not collapsed).
- **personalRoot**: `${HOME}/.claude/skills`. Always enumerated; `--scope-only=project` (or the auto-scope default below) drops personal candidates post-enumeration (see "Apply the scope filter" below).
- **projectRoots**: discovered by walking parents. **Walked unconditionally** — even with `--scope-only=personal` — so the cross-scope conflict probe below can detect cases where bare positional `<name>` resolves only in the dropped scope and produce the specific conflict message. Walk-start = `$PWD`; walk-stop = `git -C "$PWD" rev-parse --show-toplevel 2>/dev/null` (or `$PWD` if not in a git repo). Iterate from walk-start via repeated `dirname "$dir"` (quoted) until reaching walk-stop inclusive (extra safety: break if `dirname` returns the same value, i.e., at `/`).
- **Per-walked-dir filter**: for each `<dir>`, add `<dir>/.claude/skills` to `projectRoots` only if `[ -d "$dir/.claude/skills" ]` is true AND BOTH skip conditions are false. Skip if either: (a) `<dir>` is at or under `<personalRoot>` — use `realpath`-normalized paths when `REALPATH_AVAILABLE=true`, else literal trailing-slash comparison (append `/` to both sides so `/a/b` never matches `/a/bb`); or (b) `<dir>/.claude/skills` (when it exists) equals `<personalRoot>` — same `realpath`/literal switch. Condition (b) catches the case where walk-start is a *parent* of personal-root (e.g., CWD is `$HOME`) — without it, the project walk would discover personal-root as a project-root and double-count.
- **Auto-scope proposal** (computed once `projectRoots` is final, before enumeration): resolve `personalRepoRoot` = `git -C "$personalRoot" rev-parse --show-toplevel 2>/dev/null` and `cwdRepoRoot` = `git -C "$PWD" rev-parse --show-toplevel 2>/dev/null` (either may be empty — treat as "no repo"; compare the two with the same `realpath`/literal switch as the skip conditions above). Set `autoScope = project` **iff all three** hold: `cwdRepoRoot` is non-empty, `cwdRepoRoot != personalRepoRoot`, and `projectRoots` is non-empty. Otherwise `autoScope = both`. `autoScope` is a **proposal, not the applied scope** — it is consumed by "Apply the scope filter" below, which resolves the authoritative `effectiveScope`; never read `autoScope` directly at a display or reporting site (see "Resolve `effectiveScope`"). It applies **only to an unfiltered run** — an explicit `--scope-only`, a bare positional, or `--scope=<glob>` each override it, since the user has already stated the scope or the filter; auto-scope only decides the breadth of a bare `/jr-skill-audit`. Rationale: a personal-skill audit is CWD-independent (the refs cache is skill-dir-relative, `git check-ignore` runs with `-C "$personalRoot"`, and no `review-config.md` / `audit-history.json` / memory read exists), so its findings are identical wherever it runs from — auditing personal from a foreign repo duplicates a run you can do anytime, while the project scope is the only thing `$PWD` actually selects. The `projectRoots`-non-empty conjunct is a cheap early-out that skips a pointless narrow-then-unnarrow for the common no-project-skills case; it is **not** the guarantee against an auto-narrowed empty run — `[ -d "$dir/.claude/skills" ]` tests directory existence, not auditable targets, so a repo whose `.claude/skills/` exists but is empty, holds only dirs without a `SKILL.md`, or holds only gitignored skills still passes it. The **auto-narrow fallback** below is the actual guarantee.

This matches the documented Claude Code runtime behavior (per `https://code.claude.com/docs/en/skills#automatic-discovery-from-parent-and-nested-directories` — project skills load from `.claude/skills/` in the starting directory and every parent up to the repository root). Nested-on-demand discovery (e.g., `packages/frontend/.claude/skills/` from a root-started session) and `--add-dir`-mounted skills are runtime-only behaviors `/jr-skill-audit` does not have a signal for; they're out of scope here.

**Enumerate SKILL.md candidates** from each scope root, dedupe by path (using `realpath` when `REALPATH_AVAILABLE=true` to catch symlinks and any project root that resolves to personal; else literal-path dedupe — direct duplicates still caught), and tag each surviving candidate with `scope = personal | project`.

**Apply the argument-set filter** (runs BEFORE the scope filter so cross-scope conflicts can be detected; its presence/absence is also what decides whether `autoScope` applies at all):
1. **Bare positional** (`/jr-skill-audit <name>`): keep candidates whose directory basename == `<name>` in EITHER scope.
2. **`--scope=<glob>`**: keep candidates whose directory basename matches the glob in EITHER scope.
3. **No filter**: keep all candidates from both scopes. Skip directories without a `SKILL.md` (e.g., `bin/`, `docs/`, `shared/`).

**Cross-scope conflict probe** (only when `--scope-only` is set AND `<name>` is the bare positional): if the argument-filter result is **non-empty AND** every match is in the dropped scope, abort with the specific conflict message. (Empty result falls through to the generic "Available skills" hint at "Bare positional resolution" below — the empty case means `<name>` doesn't exist in EITHER scope.)
- `--scope-only=personal`, matches only in project → `Skill <name> not found in personal scope (exists in project: drop --scope-only or use --scope-only=project).`
- `--scope-only=project`, matches only in personal → `Skill <name> not found in project scope (exists in personal: drop --scope-only or use --scope-only=personal).`

**Resolve `effectiveScope`** and apply it to the argument-filter result. `effectiveScope` (`personal` | `project` | `both`) is the **authoritative record of what was actually audited** — every downstream consumer (the Phase 1 `Scope:` line, the Phase 7 `Roots:` line) reads it, never `autoScope`. Also set `autoNarrowed` (boolean): true only when auto-scope — not the user — caused a narrowing, and `autoNarrowFallbackFired` (boolean): `false` on every path except the auto-narrow fallback's success arm below, which is the sole site that sets it. An explicit `--scope-only` always wins over `autoScope`:
- `--scope-only=personal`: drop project candidates. `effectiveScope=personal`, `autoNarrowed=false`.
- `--scope-only=project`: drop personal candidates. `effectiveScope=project`, `autoNarrowed=false`.
- `--scope-only=both`: keep both. `effectiveScope=both`, `autoNarrowed=false`. Names the pre-auto-scope default explicitly — the way to audit personal + project together from a foreign repo.
- Flag absent, **argument-set filter present** (bare positional or `--scope=<glob>`): keep both. `effectiveScope=both`, `autoNarrowed=false` — **even when `autoScope=project` was computed**. The filter is the user's stated narrowing, and `<name>` must stay resolvable in either scope; auto-narrowing here would report "not found" for a skill that plainly exists (e.g. `/jr-skill-audit jr-audit` from a foreign repo).
- Flag absent, **unfiltered run**: apply `autoScope`. `project` → drop personal candidates, `effectiveScope=project`, `autoNarrowed=true`. `both` → keep both, `effectiveScope=both`, `autoNarrowed=false`.

**Collapse `both` when project scope is empty (applies to every branch above, and only here)**: once the branch resolves, if `effectiveScope=both` AND `projectRoots` is empty, set `effectiveScope=personal`. This runs once, at this point in the sequence; it is **not** re-applied to the `both` the auto-narrow fallback's abort arm sets far below (that arm is reachable only when `projectRoots` is non-empty, so the condition could not fire there anyway, but do not re-run the rule looking for it). Leave `autoNarrowed` untouched: no narrowing happened, there was simply nothing on the project side to audit. `effectiveScope` records what was *audited*, not which scopes were considered, so `both` with no project root would make the Phase 7 `Roots:` line print a `project=` segment that contributed nothing while the Phase 1 `Discovered:` line omits that same empty segment, leaving one report contradicting itself. This is reachable on the most ordinary runs (the personal skills repo itself, any non-repo dir, CWD under `personalRoot`), not only via `--scope-only=both`. It is the same invariant the auto-narrow fallback enforces at its own step 3; stating it here makes it uniform across every path instead of holding on the fallback path alone.

When `autoNarrowed=true`, retain the dropped personal candidates as `deferredCandidates` rather than discarding them — the auto-narrow fallback below may restore them. (Under an explicit `--scope-only` the drop is final: the user asserted the scope.)

**Bare positional resolution** (**only when a bare positional `<name>` was provided** — with no positional this step is inapplicable and is skipped entirely, including its 0-matches abort; final, after both filters): 0 matches → abort with the "Available skills: personal=<list> project=<list>" hint from the sanitization rule. 1 match → audit it. ≥ 2 matches → audit ALL surviving matches. The common case is exactly 2 (one personal + one project, possible only when neither `--scope-only` is set), but the parent-walk can also surface multiple project roots with the same skill name (e.g., `repo/.claude/skills/<name>` AND `repo/packages/frontend/.claude/skills/<name>` from a session started under `frontend`). The shadow-detection finding (lead-side, see "After all tracks complete" in SKILL.md) emits one `scope-resolution` finding per basename appearing in BOTH personal and project scopes; project-internal duplicates (same name across multiple walked project roots) are audited but not synthetically flagged — Claude Code's exact collision-resolution behavior for multiple walked project roots is runtime-dependent and outside `/jr-skill-audit`'s scope to specify, and the user can manually deduplicate if needed.

**Gitignore exclusion (mandatory, per scope)**: After enumeration, drop any skill whose directory is gitignored — those skills are externally maintained (e.g., installed via `npx skills`), not owned by this repo, so findings on them are not actionable here and a local fix would be clobbered on the next update. Apply per-scope independently:
Every pass below both removes the ignored skill from the target set **and appends it to `excludedCandidates`, tagged `[personal]` / `[project]`** — the field is populated here, at the site that drops them, and accumulates across every pass including the fallback's step 2. Recording is not optional bookkeeping: the Empty-discovery guard branches on `excludedCandidates` to report a cause rather than an absence, and it aborts before the discovery summary ever prints, so a pass that drops silently makes the guard claim nothing was auditable when N skills existed and were merely gitignored.
- **Personal scope**: one batched call `git -C "$personalRoot" check-ignore -- <basename>/SKILL.md ...` — every path it prints is ignored; remove from the target set. Exit 128 (`$personalRoot` not a git repo) → skip personal gitignore filtering only.
- **Project scope**: per project root R (each `<walked-dir>/.claude/skills`), one batched call `git -C "$R" check-ignore -- <basename>/SKILL.md ...`. The enclosing repo's `.gitignore` is consulted automatically via git's normal walk. Exit 128 → skip filtering for that root only.

**Bare positional** + gitignored: if the single named skill is gitignored in its only resolving scope, abort with `<name> is gitignored in <scope> (externally maintained) — /jr-skill-audit audits repo-owned skills only.` **`--scope` / no-filter**: exclude silently, but list the excluded skill names (with `[personal]` / `[project]` tags) in the Phase 1 discovery summary so the scope reduction is visible.

**Auto-narrow fallback (mandatory)**: run this AFTER gitignore exclusion — the last step that can empty the target set — and BEFORE returning to the Empty-discovery guard. **If the surviving target set is empty AND `autoNarrowed=true`**, the auto-scope proposal narrowed the run to nothing; un-narrow instead of aborting:
1. Restore `deferredCandidates` (the personal candidates dropped at the scope filter) into the target set.
2. **Run gitignore exclusion on the restored personal candidates** (accumulating any drops into `excludedCandidates` exactly as that step specifies) — they were dropped *before* the exclusion step ran, so they have never been filtered. Skipping this would let an externally-maintained skill (e.g. a gitignored `npx skills` install) into the audit through the fallback, which the exclusion rule exists to prevent.
3. Branch on whether step 2 left anything standing. **If the restored set is non-empty**: set `effectiveScope=personal`, `autoNarrowed=false`, and `autoNarrowFallbackFired=true`. **Not `both`** — the fallback fires only when the surviving set was empty, which means the project root yielded zero auditable skills, so the restored set is personal-only and `effectiveScope` must say so (it is the record of what was *audited*, not of which scopes were considered). Naming `both` here would make the Phase 7 `Roots:` line print a `project=` root that contributed nothing, while the Phase 1 `Discovered:` line omits the same empty segment — one report contradicting itself. The "no narrowing happened" signal is carried by `autoNarrowed=false` alone; `autoNarrowFallbackFired` separately drives the Phase 1 fallback `Scope:` line (see SKILL.md), which is the only output recording that the project root was tried and yielded nothing. **If the restored set is empty** (step 2 excluded every restored candidate): set `effectiveScope=both`, leave `autoNarrowed` as it stands, and do NOT set `autoNarrowFallbackFired` (no fallback survived to announce); then fall through to the abort below. **Not `personal`, and not `project`** — this is the one arm where both roots were genuinely tried and neither survived, and the abort's own message enumerates the excluded candidates from *both* scopes (`SKILL.md`, `<names with [personal]/[project] tags>`). `Roots:` must therefore name both, or the report lists a `[personal]` skill while naming only the project root and the reader cannot tell where it lives. Note this arm is the deliberate exception to `effectiveScope` meaning "what was *audited*": on an abort nothing was audited anywhere, so the field degrades to "what was tried", which is what the message is about and what the reader needs to locate the names in it.

If the set is **still** empty after the fallback (nothing auditable in either scope), fall through to the Empty-discovery guard and abort normally — the fallback must not swallow a genuine "nothing to audit" case. `effectiveScope=both` on that path is deliberate and the guard's message must stay coherent with it: the message enumerates the excluded candidates from both scopes, so `Roots:` names both roots and every `[personal]`/`[project]` name in the message has a root the reader can attribute it to.

**Abort precedence (load-bearing)**: this fallback is the **only** step permitted to act on an empty target set while `autoNarrowed=true`. No earlier abort in this protocol may pre-empt it — in particular "Bare positional resolution" above is skipped entirely on an unfiltered run, so its 0-matches abort cannot fire on an auto-narrowed empty set that this step is about to un-narrow. The two are disjoint by construction: a bare positional forces `autoNarrowed=false` at the scope filter, so the fallback never fires when that step is live, and that step never runs when the fallback is. An auto-narrowed empty set therefore reaches exactly one decision point — this one.

The fallback never fires under an explicit `--scope-only`: pinning a scope is an assertion that it exists, so `--scope-only=project` in a foreign repo with no auditable project skills still aborts. Auto-narrowing is a cost optimization and must never be a failure mode — a bare `/jr-skill-audit` that would have audited personal before this rule existed must never abort because of it.

Return the surviving target set (each tagged `scope=personal|project`), plus `effectiveScope`, `autoNarrowed`, `autoNarrowFallbackFired` (false unless the fallback above un-narrowed the run), `excludedCandidates` (every skill dropped by "Gitignore exclusion" above, accumulated across all of its passes including the fallback's step 2, each tagged `[personal]` / `[project]`), and the scope roots themselves — `personalRoot` and `projectRoots` (the parent-walk result, possibly several, possibly none) — to SKILL.md Track B, which then runs the shared "For each surviving target" read + the "Empty-discovery guard". `excludedCandidates` is both what the discovery summary's `Excluded (gitignored):` segment lists and what the Empty-discovery guard branches on to report a cause rather than an absence; that guard aborts before the summary ever prints, so the names reach it through this contract or not at all.
