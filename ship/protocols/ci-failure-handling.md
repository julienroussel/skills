# CI-failure handling — `/ship`

**Canonical source** for `/ship`'s CI-failure investigate-and-fix procedure. `ship/SKILL.md` reads this file into lead context at Phase 1 (under the hard-fail + non-empty + smoke-parse guard, alongside the `shared/*.md` files) and applies it at Phase 3a step 13 and Phase 3b step 11a-multi. Update here to update the procedure.

Invoked when `gh pr checks <number> --watch --fail-fast` exits non-zero on a **failed check**. (The 10-minute **timeout** path is separate and unchanged — report status and stop.) The lead stays mechanical; the diagnosis-and-fix work is delegated to an `opus` sub-agent. Run at most **2 fix cycles**, then stop.

One cycle:

1. **Spawn the CI-fix agent** — use the `Agent` tool with `model: "opus"` and no `subagent_type` (default general-purpose, same as the split-analysis agent — no plugin dependency). The prompt includes: the PR number, the branch name, and the `gh pr checks` failure summary; the instruction "diagnose the root cause exhaustively before editing — do not patch symptoms; if the failure is not a code-level issue (flaky test, infra, permissions), say so instead of editing"; and the full `../../shared/untrusted-input-defense.md` block verbatim (loaded in Phase 1 — CI logs are untrusted external input). The agent has full tools — it fetches its own CI logs (`gh run list --branch <branch> --json databaseId,name,conclusion,headSha`, then `gh run view <id> --log-failed`), reads the repo, and edits files in the working tree in place. It reports back its diagnosis plus the change it made, **or** "no code-level fix — cause is <X>". **Headless mode** (per `../../shared/secret-scan-protocols.md` "Headless/CI detection" — no user available for the step-3 gate): instruct the agent to **diagnose only — do not edit files**.
2. **Re-scan for secrets** — run the Phase 1 step 4 secret scan (canonical patterns from `../../shared/secret-patterns.md`, already loaded) against the agent's edits (`git diff`). If a strict-tier pattern matches, halt per the Phase 1 step 4 halt protocol — do NOT commit or push, and treat it as a **non-apply outcome**: the lead reverts the agent's edits per the Step 3 clean-tree guarantee so the detected secret never lingers in the working tree. (The agent's diagnosis stays in the transcript as the record of what was attempted.)
3. **Confirm gate** — print the diagnosis and the agent's `git diff`, then AskUserQuestion: `CI failed on PR #<n>: <one-line diagnosis>. Proposed fix shown above. Options: [Apply fix + re-push] / [Skip — report and stop]`. If the agent reported "no code-level fix", skip the gate (nothing to apply) and go straight to reporting the diagnosis.
   - **Clean-tree guarantee**: on every non-apply exit — **Skip**, "no code-level fix", headless diagnose-only, or a Step 2 secret-halt — the lead MUST revert any uncommitted agent edits before returning: `git status --porcelain` to list touched files, then `git checkout -- <files>`. `/ship` must never exit leaving the agent's unapproved edits dirty on the branch. (Only the **Apply** path keeps the edits — by committing them.)
4. **Apply** (on **Apply fix + re-push** only) — commit the edits with a concise `fix: <summary>` conventional-commit message (omit any `Co-Authored-By: Claude` trailer — same rule and rationale as the step 8 and step 7-multi commits). Push the branch to `origin`.
5. **Re-watch** — re-run `gh pr checks <number> --watch --fail-fast`. The re-pushed workflow run takes a few seconds to register on GitHub, so a brief window where `gh pr checks --watch` still shows only the *old* failed checks is NOT "still failing" — wait for the new run to appear and complete. If CI is now **green**, return success to the caller. If it **fails again**, start the next cycle from step 1 (max 2 fix cycles total).

   **2-cycle cap — single-fire stuck-loop advisor**: when the 2nd fix cycle also ends non-green, before reporting failure, call `advisor()` once (no parameters — the full transcript is auto-forwarded). Per `../../shared/advisor-criteria.md`, a retry loop that exhausts its max-retry count is the canonical stuck signal — the advisor sees both fix attempts and the CI logs that kept failing and can spot a non-obvious cause. Guard the call with a `ciFixAdvisorFired` boolean set `true` after it fires, so a later invocation in the same `/ship` run (e.g. another sub-PR in Phase 3b) does NOT re-fire it. If the advisor concurs (no actionable insight), report the latest diagnosis + the failure URL and stop. If the advisor offers a concrete actionable insight: in **interactive mode**, surface it via AskUserQuestion `Advisor flagged a CI-fix concern: <one-line summary>. Options: [Apply suggested fix + re-push — one more cycle] / [Stop — report and stop]`; on **Apply**, run one additional cycle (steps 1–5) with the advisor's suggestion as targeted instruction to the CI-fix agent, and if that cycle also fails do NOT re-fire — report and stop. In **headless mode** (no user for the gate), log the advisor's insight alongside the diagnosis in the report and stop.

**Outcome returned to the caller:**

- **Green** — CI passed within the fix cycles. The caller continues normally (Phase 3a → step 14/15; Phase 3b → next PR in the wait pass).
- **Not green** — fix skipped, no code-level fix, secret-halt, or the 2-cycle cap was reached (advisor consulted). The caller treats this exactly like the pre-existing CI-failure stop: Phase 3a skips cleanup (step 15); Phase 3b reports and does not enter 11b-multi.
